## Bug Classes Found

### Class: Untracked Background Tasks (CRITICAL)
**Pattern:** Fire-and-forget `hass.async_create_task()` calls that create long-running background tasks (with `asyncio.sleep`) that are never tracked or cancelled on teardown.

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| AC reset verify/retry tasks not tracked in `_verify_tasks` or cancelled in `teardown()`. Tasks survive coordinator teardown, wake up 30-90s later and call services against torn-down state. | `hvac_override.py` | CRITICAL | YES |
| Verify retry schedules next attempt even after service call exception, creating unbounded retry chains on persistent failures. | `hvac_override.py` | CRITICAL | YES |

**Lesson:** Every `hass.async_create_task()` that includes a sleep/delay MUST be tracked in a collection and cancelled in `teardown()`. Add this to QUALITY_CONTEXT.md.

### Class: Race Conditions Between Concurrent Reloads (CRITICAL)
**Pattern:** Options flow save triggers `async_reload()` for one entry, while simultaneously modifying another entry (triggering its reload). Shared state (`hass.data[DOMAIN]`) is deleted/recreated during unload/setup, causing KeyError or None-ref during concurrent access.

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| Room climate save calls `async_update_entry(zm_entry)` synchronously, triggering ZM reload concurrent with room reload. ZM unload deletes `zone_manager_entry` key. | `config_flow.py`, `__init__.py` | CRITICAL | YES |
| Second AC reset can fire while verify loop is still running for same zone â€” competing restore tasks. | `hvac_override.py` | CRITICAL | YES |

**Lesson:** Never modify another config entry's options from within a config flow step. Use deferred tasks or let the consumer discover state naturally. Add concurrent reload guard pattern to QUALITY_CONTEXT.md.

### Class: Broken Comparison Logic (HIGH)
**Pattern:** Smart optimization logic that compares the wrong data sources, producing incorrect results that defeat the intended optimization.

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| Smart update listener compared `entry.data` (initial config, never updated) vs `entry.options` (current options). This always finds differences for keys set during initial setup, causing unnecessary reloads. | `__init__.py` | HIGH | YES â€” simplified to: room entries skip reload entirely, non-room entries always reload |

**Lesson:** `entry.data` is immutable initial config. `entry.options` is mutable options state. HA does not provide "previous options" in update listener callbacks. Don't try to be clever about detecting which options changed â€” the simpler approach (skip reload for room entries, always reload for non-room) is both correct and safe.

### Class: Private Attribute Access Across Module Boundaries (HIGH)
**Pattern:** Code accessing `_private` attributes on objects from other modules instead of using public properties. Silently breaks if the internal attribute is renamed.

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| `_is_hvac_managing_fans()` used `getattr(mgr, '_coordinators')` and `getattr(hvac, '_fan_controller')` instead of public `.coordinators` and `.fan_controller` properties. | `automation.py` | HIGH | YES |

**Lesson:** Always use public properties for cross-module access. grep for `_coordinators` pattern when adding new cross-coordinator code.

### Class: Timezone-Naive vs Timezone-Aware Datetime Comparison (HIGH)
**Pattern:** `datetime.fromisoformat()` may produce timezone-naive datetimes when parsing serialized timestamps. Subtracting a tz-naive from a tz-aware datetime raises `TypeError`.

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| Zone state restore parsed `saved_at` with bare `datetime.fromisoformat()`. If the string lacked timezone info, comparison with `dt_util.now()` (tz-aware) would crash. | `hvac_zones.py` | HIGH | YES â€” switched to `dt_util.parse_datetime()` |

**Lesson:** Always use `dt_util.parse_datetime()` or `dt_util.as_utc()` when deserializing datetime strings in HA context. Never use bare `datetime.fromisoformat()` for timestamps that will be compared against `dt_util.now()`.

### Class: Entity State Desync with Coordinator Lifecycle (HIGH)
**Pattern:** Switch entity restores state from HA storage, but the coordinator it controls may not be initialized yet. State is applied to nothing. When coordinator initializes later, it reads from config (overwriting the switch state).

| Finding | File | Severity | Fixed |
|---------|------|----------|-------|
| Zone sweep switch restores as OFF, calls `_update_zones()`, but HVAC zones haven't been discovered yet. HVAC `async_discover_zones()` later sets `vacancy_sweep_enabled=True` from config, overriding switch. | `switch.py` | HIGH | Noted â€” MEDIUM risk accepted for now (toggle re-apply would need HVAC-ready signal listener) |

**Lesson:** RestoreEntity switches that control coordinator state need a re-apply mechanism after the coordinator finishes setup. Consider adding a dispatcher signal for coordinator-ready events.
