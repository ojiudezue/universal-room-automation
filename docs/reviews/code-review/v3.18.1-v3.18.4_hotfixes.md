## v3.18.1: @callback on 15 signal handlers

### Bug Found In Production
**Class: Missing @callback Decorator on Dispatcher Handlers (CRITICAL)**

HA's `async_dispatcher_send` runs handlers differently based on type:
- `@callback` → event loop (correct)
- Regular `def` → executor thread pool (WRONG)

15 signal handlers were missing `@callback`, causing them to run in threads. Every call to `async_schedule_update_ha_state()` from a thread raised `RuntimeError` — **2,310 occurrences** per restart.

| Handler | File | Count |
|---------|------|-------|
| `_handle_census_update` | sensor.py | 1 |
| `_handle_person_update` | sensor.py | 2 |
| `_handle_egress_event` | sensor.py | 4 |
| `_midnight_reset` | sensor.py | 2 |
| `_handle_person_update` | aggregation.py | 6 |

**Fix assessment:** Correct. Adding `@callback` is the standard HA pattern. No side effects.

**Why v3.18.0 review missed this:** The v3.18.3 thread-safety audit changed `async_write_ha_state` → `async_schedule_update_ha_state` but didn't check whether the HANDLER ITSELF needed `@callback`. The review focused on the method being called, not the calling context. This is a new bug class.

**New bug class for QUALITY_CONTEXT.md:**
- **Bug Class #22: Missing @callback on Dispatcher Signal Handlers** — All methods registered via `async_dispatcher_connect`, `async_add_listener`, `hass.bus.async_listen`, or `async_track_time_change` that write entity state MUST be `@callback` decorated. Without it, HA runs them in the executor thread pool, making all `async_write_ha_state` / `async_schedule_update_ha_state` calls thread-unsafe.

## v3.18.2: _find_zone_manager_entry on OptionsFlow

### Bug Found In Production
**Class: Method Missing on OptionsFlow Class (CRITICAL)**

`_find_zone_manager_entry()` was defined on `ConfigFlow` (line 287) but NOT on `OptionsFlow` (line 1475+). The climate step called `self._find_zone_manager_entry()` inside the OptionsFlow, raising `AttributeError` every time — 5 occurrences in the debug log.

This caused "Unknown error occurred" on the Climate & HVAC sub-menu, making it impossible to save HVAC settings.

**Fix assessment:** Correct. Added the method to OptionsFlow. Identical implementation to ConfigFlow version.

**Why v3.18.0 review missed this:** The v3.18.0 deferred-update refactor preserved the `_find_zone_manager_entry()` call without checking which class it was defined on. Both reviews (Core A and Core B) examined the deferred task logic but didn't verify the method existed on the OptionsFlow class. This is a variant of Bug Class #1 (missing imports/methods).

**Existing bug class applies:** Bug Class #1 — Missing imports/methods in secondary codepaths. Extend checklist: when refactoring code in OptionsFlow, verify all `self.*` calls resolve to methods on the OptionsFlow class, not just ConfigFlow.

## v3.18.4: DB locked, energy jitter, sensor log spam

### Bug 1: Database locked on startup
**Class: Inconsistent Connection Configuration (HIGH)**

47 logging methods in `database.py` create new `aiosqlite.connect()` calls. Only 17 had `timeout=30.0`. The other 30 used the default (5s), which is too short for 25+ concurrent room writers at startup. WAL mode and busy_timeout were set only in `initialize()` — they don't persist to new connections (per-connection pragmas).

**Fix:** New `_db()` async context manager wraps ALL connections with `timeout=30.0` + `PRAGMA busy_timeout=30000`. 69 call sites updated.

**Fix assessment:** Correct approach. The context manager pattern ensures consistency. One concern: WAL mode is set in `initialize()` but NOT in `_db()` — this is fine because WAL mode is a database-level setting that persists once set, unlike busy_timeout which is per-connection.

### Bug 2: energy_today not strictly increasing
**Class: Floating Point Precision in Cumulative Sensors (MEDIUM)**

`EnergyTodaySensor` with `state_class=TOTAL_INCREASING` reported values like 0.017788 → 0.017497 due to float arithmetic jitter. HA logs a warning for non-increasing values.

**Fix:** `round(current, 4)` before comparison — 0.1 Wh resolution eliminates sub-Wh jitter.

**Fix assessment:** Correct for the symptom. The underlying cause (float subtraction in coordinator.py energy calculation) is not addressed, but rounding at the sensor output layer is the standard approach.

### Bug 3: Sensor unavailability log spam
**Class: Excessive Logging for Transient States (LOW)**

Per-sensor "is unavailable" warnings logged on every coordinator update cycle (every 30s) for intermittently-unavailable Zigbee/Matter sensors. 134+ occurrences in a few hours.

**Fix:** Downgraded per-sensor messages from WARNING to DEBUG. Kept "All N sensors unavailable" at WARNING.

**Fix assessment:** Correct. Transient sensor unavailability is expected on wireless protocols and doesn't need WARNING-level logging.

## Process Observations

1. **4 hotfixes in one day** signals insufficient pre-deploy testing of the v3.18.0 mega-release. Future releases should deploy to a staging environment first.
2. **The @callback bug (v3.18.1) was the most impactful** — 2,310 errors destabilizing the entire integration. The v3.18.0 thread-safety audit addressed the symptom (which method to call) but missed the root cause (which thread the handler runs on).
3. **The _find_zone_manager_entry bug (v3.18.2) was a pre-existing bug** that existed before v3.18.0. Our refactoring preserved it instead of fixing it.
4. **DB locked (v3.18.4) was a pre-existing bug** that was always present but masked by the 2,310 thread errors drowning out the log.
